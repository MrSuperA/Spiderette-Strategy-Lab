# Spiderette Strategy Lab 项目开发规范

**文档版本**: v4.0.0
**最后更新**: 2026-05-07
**适用范围**: 所有 Spiderette Strategy Lab 项目开发与维护
**对应代码版本**: v5.0.5

## 📌 前言

本文档是 Spiderette Strategy Lab 项目的**强制性开发规范**，所有贡献者必须严格遵守。规范基于项目架构特性和历史经验教训（包括状态哈希缺陷、策略死循环、IntEnum 兼容性崩溃、多进程竞态等关键问题）制定，旨在确保代码质量、可维护性、安全性和可扩展性。

**违反本规范可能导致：**
- 策略决策错误（非法移动、死循环、循环检测失效）
- 统计数据失真（分布计算错误、置信区间偏差、除零崩溃）
- 多进程竞态条件（批量模拟数据污染、pickle 序列化失败）
- 状态不可逆（可变对象被意外修改，哈希链断裂）
- 打包后运行异常（路径/依赖缺失、WebView2 初始化失败）
- 未来扩展受阻（接口不统一、模块耦合过紧）

## 🏗️ 一、架构设计规范

### 1.1 四层解耦架构（核心原则）

**必须严格保持 4 层解耦，禁止跨层直接调用。** 每层只允许向下依赖，不允许向上或跨层引用。

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: UI 层 (src/ui/)                                    │
│ 职责: Flask API、pywebview 窗口、前端交互、SSE 推送          │
│ 代码: server.py + routes/ + window.py + static/             │
│ 禁止: 包含策略逻辑或游戏规则判断                             │
│ ─────────────────────────────────────────────────────────── │
│ Layer 3: 分析层 (src/analysis/)                             │
│ 职责: 量化指标、基准测试、报告生成、批量模拟、实验运行       │
│ 代码: 14 个模块（metrics/runner/batch/genetic/tournament等）│
│ 禁止: 修改核心状态或策略逻辑                                 │
│ ─────────────────────────────────────────────────────────── │
│ Layer 2: 策略层 (src/strategy/ + src/search/)               │
│ 职责: MCTS、贪心、随机等策略实现，启发式评估，搜索算法       │
│ 代码: strategy/5模块 + search/4模块                         │
│ 禁止: 直接修改 GameState，只能通过 Move 表达意图             │
│ ─────────────────────────────────────────────────────────── │
│ Layer 1: 核心层 (src/core/ + src/envs/)                     │
│ 职责: 类型定义、规则引擎、状态管理、牌局生成、模拟环境       │
│ 代码: core/6模块 + envs/3模块                               │
│ 禁止: 依赖上层模块（策略/UI/分析）                           │
└─────────────────────────────────────────────────────────────┘
```

**辅助模块（不属于四层，可被任意层引用）：**
```
┌─────────────────────────────────────────────────────────────┐
│ src/utils/     — 基础设施（paths.py / config.py / logging.py）│
│ src/rl/        — 强化学习（environment / self_play / curriculum）│
│ src/network/   — 神经网络特征（feature_v2.py）              │
│ src/iteration/ — 迭代引擎（engine.py）                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 层间通信规范

**允许的依赖方向（只能向下依赖）：**

```
Layer 4 (UI) ──→ Layer 3 (Analysis) ──→ Layer 2 (Strategy) ──→ Layer 1 (Core)
     │                  │                      │
     └──→ utils         └──→ utils             └──→ utils
     └──→ iteration     └──→ rl                └──→ rl (仅 neural.py)
```

**严格禁止：**
- Layer 1 引用 Layer 2/3/4 的任何模块
- Layer 2 引用 Layer 3/4 的任何模块
- Layer 3 引用 Layer 4 的任何模块
- 同层模块之间循环引用（如 `A imports B` 且 `B imports A`）

**已知的合法跨层引用（延迟导入）：**
- `src/core/manifest.py` → `src/strategy/registry.py`（仅在 `register_from_manifest()` 内延迟导入，避免循环）
- `src/strategy/neural.py` → `src/core/session.py` + `src/envs/simulator.py`（仅在 `train()` 方法内延迟导入，用于自博弈训练）

**新增跨层引用必须满足：**
1. 必须使用延迟导入（函数/方法内 import，非模块顶层）
2. 必须在代码注释中说明原因
3. 必须在本文档 1.2 节登记

## 📝 三、代码规范

### 3.1 Python 版本与语法

- **最低版本**: Python 3.10
- **类型注解**: 全部使用内置类型（`list[str]`、`dict[str, int]`、`X | Y`），禁止 `typing.List`、`typing.Dict`、`typing.Optional`、`typing.Union`
- **条件导入**: `tomllib`（标准库）+ `tomli`（fallback），用于 `src/__init__.py`
- **字符串格式化**: f-string 优先，复杂格式用 `.format()`
- **行长度**: 100 字符（ruff/black 配置）

### 3.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 模块文件 | snake_case | `game_session.py`、`rules_engine.py` |
| 类名 | PascalCase | `GameState`、`RulesEngine`、`MCTSNode` |
| 函数/方法 | snake_case | `legal_moves()`、`choose()`、`compute_win_rate()` |
| 常量 | UPPER_SNAKE_CASE | `COMPLETE_SEQ_LEN`、`COLUMN_COUNT` |
| 私有成员 | 单下划线前缀 | `_cache`、`_validate_move()` |
| Protocol 类 | PascalCase | `Strategy`、`Rules`、`Environment` |
| 异常类 | PascalCase + Error/Exception 后缀 | `InvalidMoveError`、`GameStateError` |

### 3.3 类型注解规范

**必须注解的场景：**
- 所有公共函数/方法的参数和返回值
- 所有 Protocol 方法
- 复杂数据结构（嵌套 dict 等）
- 回调函数签名

**示例：**
```python
# ✅ 正确
def compute_win_rate(results: list[GameResult], confidence: float = 0.95) -> tuple[float, tuple[float, float]]:
    ...

# ❌ 错误（使用旧式类型注解）
from typing import List, Optional, Tuple
def compute_win_rate(results: List[GameResult], confidence: Optional[float] = None) -> Tuple[float, Tuple[float, float]]:
    ...
```

### 3.4 不可变对象规范

**核心原则：** `GameState`、`Move`、`Card` 等核心类型必须是不可变的。

- 使用 `@dataclass(frozen=True)` 或 `__slots__` + `__hash__`
- 禁止在任何方法中修改实例属性
- 禁止返回可变内部状态的引用

### 3.5 枚举规范

- 所有枚举继承 `IntEnum`
- 禁止使用 `enum.Enum`（无法与整数比较）
- 枚举值必须从 0 或 1 开始，连续递增

### 3.6 异常处理规范

**异常层次：**
```
SpideretteError (base)
├── GameStateError          # 状态操作异常
│   ├── InvalidMoveError    # 非法移动
│   ├── GameOverError       # 游戏已结束
│   └── HashChainError      # 哈希链断裂
├── StrategyError           # 策略执行异常
│   ├── TimeoutError        # 策略超时
│   └── ResourceExhaustedError
├── ConfigError             # 配置异常
│   ├── InvalidConfigError
│   └── MissingConfigError
└── AnalysisError           # 分析异常
    ├── InsufficientDataError
    └── ComputationError
```

**使用规则：**
- 捕获具体异常，禁止裸 `except:` 或 `except Exception:`
- 异常信息必须包含上下文
- 关键路径异常必须记录日志（`_logger.error(..., exc_info=True)`）
- 资源清理使用 `try/finally` 或上下文管理器

### 3.7 文档规范

所有公共模块、类、函数必须有文档字符串。行内注释解释"为什么"而非"做什么"。

## ⚠️ 十、历史教训与反模式

### 10.1 已修复的关键缺陷

| 缺陷 | 原因 | 修复方案 | 教训 |
|------|------|---------|------|
| 状态哈希缺陷 | GameState 可变 | frozen=True dataclass | 核心类型必须不可变 |
| 策略死循环 | manifest 自引用 | base_strategy 字段 | 注册逻辑需要防环 |
| IntEnum 兼容性崩溃 | 使用 enum.Enum | 全部改用 IntEnum | 枚举必须与整数兼容 |
| 多进程竞态 | 共享可变状态 | ProcessPoolExecutor + 不可变参数 | 多进程避免共享状态 |
| LRU 缓存 FIFO | 多线程退化 | OrderedDict 手动实现 | 缓存需要线程安全 |
| 置信区间错误 | 公式实现错误 | 修正公式 | 统计公式需要单元测试 |

### 10.2 反模式清单

1. **上帝对象**: 单类超 500 行 → 拆分
2. **循环依赖**: 延迟导入或重新设计
3. **可变默认参数**: `def foo(x=[])` → `def foo(x=None)`
4. **裸异常捕获**: `except:` → `except SpecificError:`
5. **硬编码配置**: 使用 `cfg.get()`
6. **print 调试**: 使用 `logger.debug()`
7. **忽略类型注解**: 所有公共 API 必须注解
8. **跳过测试**: 新增功能必须有测试
